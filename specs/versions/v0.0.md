# v0.0 — `@actspec/expressions` standalone

**Goal:** publish the GitHub Actions expression engine as an independently useful npm package before the testing framework ships. This is the only deliverable at v0.0 — no test runner, no action execution, no mocking surface.

## Why a standalone milestone

The expression engine is the riskiest and most independently valuable component:
- **Riskiest** — "we match the C# runner" is a claim that can only be falsified by running the full conformance corpus green. That gate should be proven before the rest of the framework depends on it.
- **Independently valuable** — no complete open-source JS implementation of the GitHub Actions expression language exists. The community needs this regardless of whether they adopt actspec's test runner.
- **Pipeline proof** — publishing `@actspec/expressions` establishes the monorepo → npm publish pipeline, CI fuzz gate, and API-extractor snapshot before any further packages land.

## In scope

| Deliverable | Detail |
|---|---|
| **`@actspec/expressions`** | Full public API: `evaluate`, `evaluateTemplate`, `ExpressionError`, `ExpressionContexts` |
| Dual ESM/CJS build | Consumers can use named imports in both module systems |
| API Extractor snapshot | Public surface locked; any breaking change fails CI |
| Full corpus green | All 459 vectors in `corpus/expressions/` pass |
| Parser/eval fuzz in CI | Differential fuzz against the documented grammar; non-blocking vs `nektos/act` oracle |
| `hashFiles` real implementation | Glob + SHA-256 per the runner spec; overridable via the `functions` hook |

## Out of scope (all belong to v0.1+)

- `@actspec/core`, `@actspec/composite`, or any other package
- The test runner, CLI, matchers, coverage
- The conformance corpus for protocol / runner behavior (those pin runner fidelity, not expression fidelity)

## Gate (v0.0 is "done")

1. `@actspec/expressions` published to npm.
2. Full corpus green: all 459 vectors in `corpus/expressions/` pass — including the complete `&&`/`||` type matrix and all function test cases from `nektos/act`.
3. Parser/eval fuzz in CI (green, non-blocking vs act oracle).
4. API Extractor snapshot committed; breaking-surface CI check passing.
5. Dual ESM/CJS build verified (`@arethetypeswrong` clean).

## Corpus status at v0.0

The corpus is **fully harvested** — 459 vectors across 7 files, mechanical harvest from `nektos/act`'s `interpreter_test.go` + `functions_test.go`. See [corpus/expressions/PROVENANCE.md](../../corpus/expressions/PROVENANCE.md).

One recorded divergence: `format('{2147483648}')` — act/runner says "invalid format string" (int32 overflow); our evaluator falls through to "references more arguments than were supplied". Both throw; to be aligned with runner behavior when the package is built.

## Implementation note

The expressions spike in `spike/expressions/` produced a working Pratt parser and evaluator. **Do not rewrite from scratch** — copy and evolve that code into `@actspec/expressions`. The spike code is the starting point; the module spec and corpus are the finishing gate.
